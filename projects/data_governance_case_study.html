Data Governance & a Unified Semantic Model

Establishing the governance framework behind the portfolio's analytics — a single, trusted semantic model with row-level security, an audit-ready data dictionary, documented data flows, and a monthly governance rhythm that holds external managing agents accountable for data quality.

Context

A trusted reporting platform needs more than good pipelines — it needs governance: clear definitions, controlled access, documented lineage, and accountability for the quality of incoming data. With data arriving from multiple internal teams and several external managing agents, inconsistent submissions were a constant risk to the integrity of every report.

I established the governance framework that wraps around the analytics — defining the model, the rules, the documentation and the operating rhythm that keep the numbers dependable.

The challenge

  • Inconsistent definitions. The same metric meant different things to different teams, eroding trust.
  • Variable data quality. External agents submitted data to different standards, completeness and timeliness.
  • Access control. Audiences needed tailored views without seeing data outside their remit.
  • Limited documentation. Business logic lived in people's heads, not in an auditable reference.

What I built

1 · One governed semantic model

A single Power BI semantic model is the canonical definition of every metric. Measures are written once in DAX and reused across all report areas — financial performance, lettings, arrears, rent changes, compliance, EDLP and complaints — so figures reconcile by construction rather than by reconciliation.

2 · Row-level security & audience-based access

The model is distributed through a governed Power BI app. Row-level security and audience-based access ensure each team sees exactly the slice relevant to them — the same trusted source, tailored and secured per role.

3 · An audit-ready data dictionary

I built a full data dictionary documenting every table, column and Power BI metric, including how each measure is calculated. Designed for audit, it turns the model from a "black box" into a transparent, self-describing asset — and dramatically lowers the barrier for new team members and report builders.

Documented lineage. Process documentation and end-to-end data flow diagrams map data from source submission through to the metric on the report — so anyone can see where a number comes from.

4 · A monthly governance rhythm

I lead monthly data governance meetings with external managing agents — setting submission standards, running the agenda, and holding agents accountable for the quality and completeness of what they provide. Automated validation built into the pipeline flags issues early, turning data quality into a managed, visible process rather than a surprise at month-end.
